    1. Read existing `.github/workflows/ci.yml`. Preserve the existing `test` job verbatim.

    2. Append a new `install-test` job under `jobs:` (sibling to `test`). Full YAML to add:

       ```yaml
         install-test:
           name: Fresh-VM install (${{ matrix.os }})
           runs-on: ${{ matrix.os }}
           strategy:
             fail-fast: false
             matrix:
               os: [ubuntu-latest, macos-latest, windows-latest]
           steps:
             - uses: actions/checkout@v4

             - name: Set up Python 3.11
               uses: actions/setup-python@v5
               with:
                 python-version: "3.11"

             - name: Install package (simulates `pip install sekha` on fresh VM)
               run: pip install -e .

             - name: Verify sekha console script is on PATH
               shell: bash
               run: |
                 which sekha || command -v sekha
                 sekha --help

             - name: Run sekha init
               shell: bash
               run: |
                 sekha init
                 echo "--- ~/.sekha/ tree ---"
                 ls -la "$HOME/.sekha" || dir "%USERPROFILE%\\.sekha"
                 echo "--- ~/.claude/settings.json ---"
                 cat "$HOME/.claude/settings.json" || type "%USERPROFILE%\\.claude\\settings.json"

             - name: Run sekha doctor (JSON mode) and assert critical checks pass
               shell: bash
               run: |
                 set -euo pipefail
                 sekha doctor --json > doctor.json
                 cat doctor.json
                 python - <<'PY'
                 import json, sys
                 data = json.loads(open("doctor.json").read())
                 checks = {c["name"]: c for c in data["checks"]}
                 required = [
                     "python_version",
                     "sekha_on_path",
                     "sekha_home_writable",
                     "settings_hook_registered",
                     "mcp_canary",
                     "kill_switch",
                 ]
                 failures = [n for n in required if not checks.get(n, {}).get("ok")]
                 if failures:
                     print(f"FAIL: required checks not OK: {failures}", file=sys.stderr)
                     for n in failures:
                         print(f"  {n}: {checks.get(n, {}).get('detail', '<missing>')}", file=sys.stderr)
                     sys.exit(1)
                 print("All required doctor checks passed.")
                 PY

             - name: Run sekha init again (idempotency smoke)
               shell: bash
               run: |
                 set -euo pipefail
                 sekha init
                 python - <<'PY'
                 import json, os
                 p = os.path.expanduser("~/.claude/settings.json")
                 data = json.loads(open(p).read())
                 count = 0
                 for entry in (data.get("hooks") or {}).get("PreToolUse") or []:
                     for h in entry.get("hooks") or []:
                         if h.get("command") == "sekha hook run":
                             count += 1
                 assert count == 1, f"expected exactly 1 sekha hook entry, got {count}"
                 print("Idempotency OK: exactly one sekha hook entry.")
                 PY
       ```

       Key design decisions:
       - `shell: bash` on every step — Windows runners ship with Git Bash, so this works uniformly across the 3 OSes and avoids PowerShell/cmd escaping hell.
       - Python 3.11 only in this job (not 3×3) — the `test` job already covers the full version matrix. The install-smoke only needs one Python version per OS to prove the CLI-08 release gate.
       - `set -euo pipefail` in multi-line shell blocks so silent failures are caught.
       - The assertion script is embedded via heredoc (`python - <<'PY' ... PY`) — no extra scratch files, pure stdlib, mirrors the rest of Sekha's stdlib-only discipline.
       - `fail-fast: false` mirrors the existing `test` job — failures on one OS shouldn't mask the others.
       - No `needs:` clause — the install-test is independent of the unit-test job and they should run in parallel.
       - The fallback `|| dir ...` and `|| type ...` branches in the `ls` / `cat` steps are defensive: if bash resolution of `$HOME` ever fails on Windows, we still get a human-readable listing in the log.

Note: The ROADMAP specifies the fresh-VM install uses `pip install <name>` (i.e., from PyPI). Because v0.0.0 has not yet been published to PyPI (deferred from Plan 00-02 per 06-CONTEXT.md), we substitute `pip install -e .` — functionally equivalent for the install-smoke goal (simulates pip installing the package into a clean environment) and this is exactly what the CONTEXT.md specifies. When Phase 7 ships v0.1.0 to PyPI, a follow-up plan can switch to `pip install sekha==<version>` in a tag-gated second job.

Not covered by this plan (out of scope for CLI-08, covered elsewhere):
- `claude mcp add sekha -- sekha serve` is NOT run in CI because the GitHub runner has no Claude Code installed. The MCP canary inside `sekha doctor` is the proxy for that check — it spawns `sekha serve` as a subprocess and performs the JSON-RPC initialize handshake. That proves the server works end-to-end without requiring Claude Code in CI.
